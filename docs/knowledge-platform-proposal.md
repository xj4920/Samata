# 衍生品展业知识沉淀与AI赋能方案

---

## 一、现状与痛点

### 1.1 知识散落各处

团队在衍生品展业过程中积累了丰富的实战经验，但这些知识分散在群聊、邮件、文档、口头沟通等多个渠道中，碎片化严重。同一个问题反复被不同客户问到，每次都需要从头组织回答，效率低下。

### 1.2 分支客户激增，专业知识未对齐

随着业务拓展，越来越多的分支机构和新客户接入。客户技术水平参差不齐，不少客户需要从最基本的概念开始了解。FIX 协议对接、北上极速接入、算法单配置等问题被反复解答，团队在重复劳动上消耗了大量精力。

### 1.3 衍生品及极速业务专业门槛高

衍生品业务本身专业术语密集——FIX Protocol、极速交易、算法单、券池、估值、风控等概念交织。跨系统对接更涉及大量技术细节：接入方式选择、重连机制设计、延迟优化、日志排查等。目前缺乏结构化的知识体系，新人和客户都难以快速理解全貌。

## 二、解决方案：OTCClaw

### 2.1 背景与思路

团队深入衍生品展业已有一段时间，在日常业务沟通中积累了大量实战经验和专业知识。与此同时，AI 大模型能力日趋成熟，在知识梳理、智能问答、辅助决策等方面已具备实用价值。

核心思路：**把散落的业务经验沉淀为结构化知识，用 AI 放大展业效率。**

### 2.2 为什么借鉴 xxxClaw 架构

选择它的原因：

- **工具自举（Tool Bootstrapping）**：工具可以调用工具，形成自动化链路，减少人工干预
- **Channel 通信**：通过通道机制操作服务端资源、执行命令，灵活性强
- **轻量安全**：本地部署，数据不出内网；SQLite 存储，无需重型数据库，运维成本极低
- **高度可定制**：从0到1编码，形成了我们自己的产品——**OtcClaw（衍语）**

### 2.3 OtcClaw（衍语）系统架构

OtcClaw（衍语）是深度定制的客户展业全生命周期管理系统，为销售团队提供从初次接触到上线投产的全流程客户跟踪，内置 AI 助手辅助日常操作。

#### 技术选型


| 层级    | 技术                            | 说明               |
| ----- | ----------------------------- | ---------------- |
| 语言    | TypeScript + Node.js          | 类型安全，开发效率高       |
| 存储    | SQLite（WAL 模式）                | 轻量、零运维、本地化       |
| AI 引擎 | Anthropic Claude SDK + 多模型适配  | 主力模型 + 按场景切换     |
| 交互    | Inquirer.js + 飞书/Telegram SDK | CLI 标准实现 + 多渠道扩展 |


#### 模块化架构

系统采用模块化设计，各模块职责清晰、松耦合：

```
src/
├── index.ts          # 入口，交互式 REPL
├── auth/             # 权限控制（RBAC）
├── commands/         # 命令处理器（业务逻辑层）
├── db/               # 数据库连接与 Schema
├── llm/              # AI Agent（工具定义 + Agentic Loop）
├── models/           # 数据模型
├── plugins/          # 插件系统（可扩展架构）
├── services/         # 外部服务集成（飞书、Telegram、消息感知）
├── scripts/          # 数据导入工具
└── utils/            # 日志、表格渲染等通用工具
```

核心设计原则：**命令层（commands/）承载所有业务逻辑，AI Agent（llm/）通过工具调用复用命令层能力，所有渠道（CLI / 飞书 / Telegram）共享同一套 Agent 逻辑**，确保行为一致、维护成本低。

#### 客户生命周期模型

系统内置 5 阶段客户流转模型，覆盖展业全流程：

```
初次接触 → 需求沟通 → 方案设计 → UAT 测试 → 上线投产
```

每个阶段支持状态跟踪、操作审计、阶段推进，配合 15+ 命令实现完整的客户管理闭环。

#### 交互路由逻辑

系统对用户输入采用统一的路由策略，三个渠道（CLI / 飞书 / Telegram）遵循相同逻辑：

```
                        ┌─── CLI 命令行 ───┐
                        │                   │
         用户输入 ◄─────┤─── 飞书机器人 ───┤
                        │                   │
                        └─── Telegram ─────┘
                                │
                                ▼
                    ┌───────────────────────┐
                    │     输入类型判断       │
                    └───────────────────────┘
                         │            │
                    "/" 命令        自然语言
                         │            │
                         ▼            ▼
              ┌─────────────┐  ┌──────────────────┐
              │  命令路由器   │  │  AI Agent        │
              │  (router)    │  │  runAgenticChat()│
              └──────┬──────┘  └────────┬─────────┘
                     │                  │
                     ▼                  ▼
              直接执行命令        LLM 理解意图
              无需 AI 介入        自主规划工具调用
                     │                  │
                     │                  ▼
                     │         ┌────────────────┐
                     │         │  40+ 内置工具   │
                     │         └───┬────┬───┬───┘
                     │             │    │   │
                     ▼             ▼    ▼   ▼
              ┌──────────────────────────────────┐
              │          命令层 (commands/)        │
              │                                   │
              │  客户管理  交易查询  知识库检索     │
              │  技能管理  文件操作  记忆存取  ...  │
              └──────────────┬───────────────────┘
                             │
                             ▼
              ┌──────────────────────────────────┐
              │          数据层                    │
              │  SQLite（客户/知识库/审计日志）     │
              │  InfluxDB（交易数据）              │
              └──────────────────────────────────┘
```

两条路径的区别：


|          | "/" 命令（直接执行）             | 自然语言（AI Agent）             |
| -------- | ------------------------ | -------------------------- |
| 示例       | `/faq FIX协议`、`/trade 查询` | "帮我查一下磐松最近的交易情况"           |
| 是否调用 LLM | 否                        | 是                          |
| 执行方式     | 命令路由器直接分发到对应命令函数         | LLM 理解意图 → 自主选择工具 → 调用命令函数 |
| 适用场景     | 明确知道要做什么，快速操作            | 模糊需求、复杂查询、多步骤任务            |
| 工具组合     | 单一命令                     | AI 可串联多个工具协作完成             |


关键设计：**AI Agent 的工具调用最终复用的是同一套命令层函数**，不存在重复实现。无论用户通过哪个渠道、哪种方式交互，底层业务逻辑只有一份。

### 2.4 核心功能介绍

#### 知识库管理

系统内置结构化 FAQ 知识库，支持：

- 多关键词拆分 + 相关性评分的智能搜索，快速定位答案
- 按主题分类管理，覆盖 FIX 协议、接入方式、北上极速、券池、费用等 15+ 业务主题
- 知识条目持续积累，形成团队专属的专业知识体系

#### AI Agent 多工具协作

系统搭载 AI Agent，具备 30+ 内置工具，涵盖交易查询、客户管理、知识检索、业务沟通记录分析等能力。Agent 采用 Agentic Loop 模式运作：AI 自主理解问题 → 规划调用路径 → 执行工具 → 整合结果 → 输出回答。支持多模型切换（Claude / Gemini / MiniMax / OpenRouter），可按场景选择最合适的模型。

#### 多渠道统一接入

为适应不同使用场景，系统支持多渠道接入：

- **CLI 命令行**：标准实现，功能最完整
- **飞书机器人**：支持 WebSocket/Webhook 接入，卡片消息展示，思考过程可视化
- **Telegram 机器人**：移动端便捷访问

所有渠道共享同一套 Agent 逻辑，行为完全一致，避免多端维护的不一致问题。

#### 客户全生命周期管理

- 客户信息管理与状态跟踪
- 交易数据查询与曲线图生成，直观展示业务数据
- 业务群消息动态感知与多通道通知，确保重要信息不遗漏

#### 安全与权限

- **RBAC 角色控制**：区分管理员与普通用户权限，敏感操作需管理员授权
- **数据本地化**：SQLite 本地存储，数据全程不离开服务器，不经过任何外部服务
- **操作可审计**：关键操作留有日志，可追溯

### 2.5 QA-Forge：知识提炼工具

QA-Forge 是一个辅助工具，用于从团队日常业务沟通记录中梳理、归纳结构化知识。

#### 智能梳理

- 按 15+ 预定义业务主题，从历史沟通记录中归纳常见问题与标准答案
- 增量处理，自动去重，避免重复归纳
- 滑动窗口机制，确保上下文完整性

#### 质量保障

- 语义去重，相似问题自动合并
- 确保知识条目准确、无冗余

#### 人工审核流程

**所有知识条目均经人工逐条审核后方可入库**，确保内容准确性与合规性。审核过程中支持编辑修正，完整审计日志保证所有操作可追溯。

#### 整体流程

```
业务沟通经验 → AI 辅助梳理 → 去重评分 → 【人工审核】 → 正式知识库 → AI Agent 可检索
```

## 三、下一步计划

### 3.1 打通企微入口

接入企微机器人，员工在企微群内直接 @bot 即可提问，复用现有 Agent 逻辑（与 CLI、飞书一致），与现有业务沟通场景无缝衔接。

### 3.2 内部试用

先在团队内部试运行，收集使用反馈。持续补充知识库——定期梳理业务经验，经人工审核后入库。根据实际使用情况持续优化。

### 3.3 远期展望

- 知识库覆盖更多业务主题，形成更完整的衍生品业务知识图谱
- 开放客户自助查询入口，降低团队重复解答负担
- 与内部系统（CRM、交易系统）更深度集成，打通数据孤岛
